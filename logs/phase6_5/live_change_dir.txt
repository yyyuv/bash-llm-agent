TEST 4 — REAL terminal confirmation (not a sandboxed zsh -c test), + a new bug found (P6.5d)
Raw log: logs/phase6_5/p6_5.jsonl

First run, in a terminal predating the ~/.zshrc edit:
  `type doit` -> "doit is /Users/yuvalreuveni/.local/bin/doit" (raw binary,
  wrapper not loaded). doit "go to logs/ folder" left pwd unchanged --
  expected, since without the doit() function the binary alone can never
  touch the parent shell (this IS D1's point, not a bug).

After `source ~/.zshrc`:
  `type doit` -> "doit is a shell function from /Users/yuvalreuveni/.zshrc"
  $ doit "go to logs/ folder"
  · will cd to /Users/.../bash-llm-agent/logs after this run
  $ ls
  acdl  CLAUDE.md  DECISIONS.md  doit  doitlib  logs  PLAN_DETAILED.md
  PLAN.md  prompts  sessions  shell  tests
  yuvalreuveni@yuvals-MacBook-Pro logs %
Result: PASS on the core mechanism -- the zsh PROMPT ITSELF changed from
"bash-llm-agent %" to "logs %", the strongest possible confirmation since
the prompt is generated by the live shell, not by doit.

BUT: the request only asked to go to logs/ -- nothing about listing -- yet
the model still ran an unrequested `ls`, which (per P6.5b) executed before
the real cd and printed the WRONG (old) directory's contents. This is a
broader trigger than test 3's explicit "and list it" case: here the model
volunteered the follow-up entirely on its own, so P6.5b's fix (which only
tells the model how to target the path IF it adds a follow-up) didn't stop
the follow-up from happening. See DECISIONS.md P6.5d.

Fix: strengthened both the change_dir tool schema and its returned
observation text: do NOT add a follow-up command unless the user
explicitly asked to see/list the directory; otherwise answer and stop.
